# Loafer Blog 功能增强 - 产品需求文档

## Overview
- **Summary**: 对Loafer Blog系统进行功能增强，包括数据库表优化、全局配置管理、文件大小限制动态配置、业务RSA密钥对管理、邮箱加密存储、默认头像生成和添加朋友功能。
- **Purpose**: 提升系统的安全性、可配置性和用户体验。
- **Target Users**: 所有Loafer Blog用户，包括普通用户和管理员。

## Goals
- 设置所有表的delete_time字段默认值为'00001-01-01 00:00:00'
- 添加全局配置表，支持系统配置管理
- 实现文件大小限制的动态配置
- 增加业务RSA密钥对管理，用于敏感数据加密
- 修改用户注册功能，使邮箱为非必填项并加密存储
- 在个人资料页面增加邮箱地址的回显与修改功能
- 修复发送消息时sender_id为空的错误
- 为用户生成默认头像
- 增加添加朋友功能，支持用户搜索和分页

## Non-Goals (Out of Scope)
- 不修改系统的核心架构
- 不改变现有的用户认证机制
- 不添加新的第三方依赖

## Background & Context
- Loafer Blog是一个现有的博客系统，使用Spring Boot + MyBatis Plus + PostgreSQL
- 系统已实现基本的用户管理、文章发布、消息等功能
- 前端使用Vue.js框架

## Functional Requirements
- **FR-1**: 设置所有表的delete_time字段默认值为'00001-01-01 00:00:00'
- **FR-2**: 添加全局配置表，包含主键、模块、key、value、描述等字段
- **FR-3**: 文件大小限制相关接口改为专用DTO接收返回参数
- **FR-4**: 将用户修改后的文件大小限制配置存储到配置表中
- **FR-5**: 根据配置表中的值动态修改web应用的上传文件大小限制
- **FR-6**: 项目启动时从配置表中读取文件大小限制配置
- **FR-7**: 增加一对可持久化的业务RSA密钥对文件
- **FR-8**: 在application.yml中配置业务RSA密钥对存放路径
- **FR-9**: 项目启动时检查业务RSA密钥对文件是否存在，不存在则生成并存储
- **FR-10**: 将业务RSA密钥对添加到全局缓存
- **FR-11**: 注册功能邮箱改为非必填项
- **FR-12**: 前端使用通信RSA公钥加密邮箱后发送给后端
- **FR-13**: 后端使用通信RSA私钥解密邮箱并校验格式
- **FR-14**: 注册信息检验通过后使用业务RSA公钥加密邮箱并保存到数据库
- **FR-15**: 调整user表email字段长度，防止密文超长保存失败
- **FR-16**: 个人资料页面增加邮箱地址的回显与修改功能
- **FR-17**: 后端从user表查询出的邮箱字段为业务RSA公钥加密后的密文，需要先使用业务RSA私钥解密后，再进行脱敏返回给前端
- **FR-18**: 修改个人资料时，前端传参如果发现该字段未修改（或者说是脱敏后的格式），那么不传递该参数给后端
- **FR-19**: 如果修改了邮箱，并且前端邮箱格式验证通过，那么使用通信RSA公钥加密传输给后端
- **FR-20**: 后端验证邮箱字段格式，如果字段不为空，且不是有效的通信RSA公钥加密密文，则跳过该字段的更新
- **FR-21**: 如果是有效密文，先使用通信RSA私钥解密，再验证邮箱格式，保存到数据库时使用业务RSA公钥加密存储
- **FR-22**: 修复发送消息功能报错：错误: null value in column "sender_id" of relation "message" violates not-null constraint
- **FR-23**: 用户注册时，后端自动生成一张随机颜色的纯色背景图片，填充用户名首字母为图片主体内容
- **FR-24**: 字体颜色要和背景色有明显区分
- **FR-25**: 将生成的图片保存到头像上传目录（如果目录不存在，则创建）
- **FR-26**: 生成访问URL地址保存到user表中
- **FR-27**: 项目启动时查询所有头像地址为空的user记录，按照此逻辑为他们依次生成默认头像
- **FR-28**: 消息页增加一个添加朋友的功能
- **FR-29**: 可以通过用户名搜索本站的所有用户
- **FR-30**: 分页查询，按照用户ID排序
- **FR-31**: 滑动到页面底部时，显示"加载更多"按钮
- **FR-32**: 查询下一页数据需要前端传递上一页的最大用户ID给后端
- **FR-33**: 后端添加用户ID大于此参数的查询条件，避免深度分页问题

## Non-Functional Requirements
- **NFR-1**: 系统安全性高，敏感数据使用RSA加密存储
- **NFR-2**: 系统可配置性强，支持动态修改文件大小限制
- **NFR-3**: 系统性能良好，分页查询避免深度分页问题
- **NFR-4**: 用户体验良好，默认头像生成美观
- **NFR-5**: 代码可维护性好，结构清晰

## Constraints
- **Technical**: 基于现有技术栈（Spring Boot + MyBatis Plus + PostgreSQL）进行开发
- **Business**: 保持系统稳定性，不影响现有功能
- **Dependencies**: 依赖现有的系统架构和组件

## Assumptions
- 现有系统架构和数据库结构基本合理，不需要大的调整
- 服务器环境正常，网络连接稳定
- 前端和后端代码结构清晰，便于扩展和修改
- 系统已有通信RSA密钥对用于前端参数加密

## Acceptance Criteria

### AC-1: delete_time默认值设置
- **Given**: 查看数据库表结构
- **When**: 检查表的DDL语句
- **Then**: 所有表的delete_time字段默认值为'00001-01-01 00:00:00'
- **Verification**: `programmatic`

### AC-2: 全局配置表创建
- **Given**: 查看数据库结构
- **When**: 检查是否存在配置表
- **Then**: 配置表已创建，包含主键、模块、key、value、描述等字段
- **Verification**: `programmatic`

### AC-3: 文件大小限制动态配置
- **Given**: 访问文件大小限制配置接口
- **When**: 修改文件大小限制
- **Then**: 配置被保存到配置表，且系统上传限制动态更新
- **Verification**: `programmatic`

### AC-4: 业务RSA密钥对管理
- **Given**: 项目启动
- **When**: 检查业务RSA密钥对文件
- **Then**: 如果文件不存在则生成，存在则读取并添加到全局缓存
- **Verification**: `programmatic`

### AC-5: 邮箱加密存储
- **Given**: 用户注册
- **When**: 提交注册信息
- **Then**: 邮箱（如果提供）被加密存储到数据库
- **Verification**: `programmatic`

### AC-6: 个人资料邮箱回显与修改
- **Given**: 用户进入个人资料页面
- **When**: 查看个人信息
- **Then**: 邮箱以脱敏形式显示，且可以修改
- **Verification**: `programmatic`

### AC-7: 发送消息功能修复
- **Given**: 用户发送消息
- **When**: 提交消息
- **Then**: 消息发送成功，无sender_id为空错误
- **Verification**: `programmatic`

### AC-8: 默认头像生成
- **Given**: 用户注册或项目启动
- **When**: 检查用户头像
- **Then**: 为无头像用户生成默认头像
- **Verification**: `programmatic`

### AC-9: 添加朋友功能
- **Given**: 用户进入消息页面
- **When**: 使用添加朋友功能
- **Then**: 可以搜索用户并分页加载，点击加载更多按钮获取下一页数据
- **Verification**: `programmatic`
